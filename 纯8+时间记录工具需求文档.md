**开发顺序**（按4.4节的开发路径）：

| 阶段 | 任务 | 参考章节 |

**关键约束**（基于Linus Torvalds建议）：

```
1. 简单胜过聪明 → 数据结构扁平，避免过度抽象
2. 显式胜过隐式 → 状态变化显式触发，不用魔法响应
3. 代码要能活二十年 → 十年后陌生人能看懂
```

**代码风格要求**：

| 要求 | 说明 |

| **错误处理** | 关键操作必须有try-catch和用户友好提示 |

## 2.PRD详情内容

### 2.1 背景与目的

#### 2.1.1 产品背景

**用户痛点**：

1. **虚无感的困扰**：投入大量时间学习/成长，但回望时感觉"我似乎什么都没做"，大脑的"缩略图效应"让人产生虚度感
2. **惰性难以克服**：计划总是半途而废，"flag反复立反复倒"
3. **缺乏即时反馈**：努力是隐性的，无法立刻看到成果，容易失去动力
4. **自我效能感低**：看不到自己的进步，缺乏成就感和征服的快感

**核心理念来源**：

- **纯8+理念**：每天精确计时达到纯8小时以上的高质量学习时间，让时间投入"可见、可信、可累积"
- **画格子方法**：把任务变成可视化格子，贴在墙上，完成一个划掉一个，让努力看得见
- **三次打卡法**：同一件任务，让自己爽三次（划格子+记录+展示），最大化自我效能感

> **本期MVP聚焦：纯时间记录 + 画格子可视化反馈**
> 时间轮功能（输入输出配比、轮次概念）留待后续迭代

#### 2.1.2 产品目的

**宏观目的**：
> **通过记录时间，看见时间的力量，对抗虚无与自身惰性，获得征服的快感，提高自我效能感。**

**具体目的**：

1. **让时间付出可视化**：像守财奴存钱一样，每一分钟的投入都被记录、被看见，成为永久的"财富"
2. **提供即时正向反馈**：通过画格子的外显方式，让每次任务完成都能立刻获得"划掉一个格子"的爽感
3. **培养长期自律习惯**：从"被迫坚持"到"享受坚持"，通过记录证明自己的能力
4. **帮助用户认清自己**：通过真实数据找到适合自己的"纯数字"，不盲目对标他人

#### 2.1.3 产品定位

> **一款让时间努力"看得见"的个人成长时间记录工具**

- 不是复杂的任务管理工具
- 不是日程安排工具
- 而是：**纯时间记录 + 画格子可视化**，两个独立功能的组合

### 2.6 详细功能需求

#### 2.6.1 页面布局：底部Tab导航

**整体结构**：应用采用底部Tab导航，分为两个独立的Tab页面。

```
┌─────────────────────────────────┐
│                                 │
│                                 │
│         [主内容区域]              │
│      (时间记录 OR 画格子)         │
│                                 │
│                                 │
├─────────────────────────────────┤
│  ⏱ 时间记录    ▦ 画格子         │
└─────────────────────────────────┘
```

| Tab | 图标 | 名称 | 内容 |
|-----|------|------|------|
| 左侧 | ⏱ 秒表时钟 | 时间记录 | 纯粹的秒表计时功能 |
| 右侧 | ▦ 格子图标 | 画格子 | 手动填满格子的可视化反馈 |

**关键设计**：
- 两个Tab完全独立，互不干扰
- 用户自由切换，没有强制关联
- 界面简洁，每个Tab只做一件事

#### 2.6.2 Tab 1：时间记录

**功能描述**：纯粹的秒表计时功能，记录用户投入的时间。

**核心理念**：
> **时间记录是纯粹的计时功能，与画格子项目完全独立。**
> **不关联任何项目，不预设目标，只是记录你投入了多长时间。**

**界面设计**：

```
┌─────────────────────────────────┐
│         ⏱ 时间记录               │
├─────────────────────────────────┤
│                                 │
│       [ 大圆圈计时器 ]           │
│         00:25:30                │
│         (时:分:秒)               │
│                                 │
│     [    开始计时    ]           │
│                                 │
├─────────────────────────────────┤
│ 今日记录：                       │
│ • 09:00-10:30  1h30m            │
│ • 14:00-15:15  1h15m            │
│ • 20:00-21:00  1h0m             │
│                                 │
│  ────────────────────────       │
│  今日累计：4小时15分钟            │
│                                 │
│  ────────────────────────       │
│  今日目标：2.5小时 (可选)        │  ← "我的数字"设置后显示
│  完成度：171%                   │
│                                 │
│  ────────────────────────       │
│  📊 查看详细统计  →              │  ← 点击跳转到统计页
└─────────────────────────────────┘
```

**说明**：
- "今日目标"仅在用户通过"我的数字"设置目标后才显示
- 未设置目标时，不显示目标部分

**页面元素**：

| 元素 | 说明 | 交互 |
|-----|------|------|
| **大圆圈计时器** | 显示当前计时时长，精确到秒 | 计时时实时更新 |
| **开始/停止按钮** | 计时的核心控制按钮 | 点击开始/停止计时 |
| **今日记录列表** | 显示今日所有计时记录 | 按时间倒序排列 |
| **今日累计** | 显示今日总投入时间 | 实时计算所有记录的总和 |
| **查看详细统计入口** | 点击跳转到统计页 | 查看"我的数字"等详细统计 |

**计时逻辑**：

1. **开始计时**：
   - 点击"开始计时"按钮
   - 大圆圈开始显示计时（00:00:00 → 00:00:01 → ...），**精确到秒**
   - 按钮文字变为"停止计时"

2. **停止计时**：
   - 点击"停止计时"按钮
   - 计时暂停，显示最终时长
   - 自动保存到"今日记录"列表
   - 更新"今日累计"时间

3. **再次开始**：
   - 可以多次开始/停止
   - 每次停止都会生成一条新记录
   - 记录列表显示所有计时片段

**时间精度说明**：
- **计时阶段**：大圆圈显示精确到秒（00:25:30）
- **记录列表**：显示精确到分钟（1h30m）
- **统计页面**：显示精确到分钟（4小时15分钟）

// 时间记录
interface TimeRecord {

#### 2.7.3 关键技术点

**1. 时间精度实现**

```typescript
// ❌ 错误：使用系统时间（会被用户修改影响）
const start = Date.now();

// ✅ 正确：使用单调时钟（浏览器环境）
// 注意：浏览器没有原生的单调时钟API，需要用workaround
class Stopwatch {
  private startTime: number;
  private elapsedBeforePause: number = 0;

  start() {
    this.startTime = performance.now();
  }

  pause() {
    this.elapsedBeforePause += performance.now() - this.startTime;
  }

  get elapsed(): number {
    return this.elapsedBeforePause + (performance.now() - this.startTime);
  }
}
```

**重要说明**：
- `performance.now()` 是高精度计时器，不受系统时间调整影响
- 但它只在页面运行时有效，页面关闭后会清零
- 存储时记录 `start_timestamp` 和 `duration`，恢复时从timestamp计算

**2. Canvas绘画实现**

```typescript
class GridCanvas {
  private canvas: HTMLCanvasElement;
  private ctx: CanvasRenderingContext2D;
  private points: {x: number, y: number}[] = [];
  private maxPoints = 200; // 限制笔触数量

  constructor(canvas: HTMLCanvasElement) {
    this.canvas = canvas;
    this.ctx = canvas.getContext('2d')!;
    this.setupEvents();
  }

  private setupEvents() {
    let isDrawing = false;

    this.canvas.addEventListener('pointerdown', (e) => {
      isDrawing = true;
      this.addPoint(e.offsetX, e.offsetY);
    });

    this.canvas.addEventListener('pointermove', (e) => {
      if (!isDrawing) return;
      this.addPoint(e.offsetX, e.offsetY);
      this.redraw();
    });

    this.canvas.addEventListener('pointerup', () => {
      isDrawing = false;
    });
  }

  private addPoint(x: number, y: number) {
    if (this.points.length >= this.maxPoints) {
      return; // 达到上限，不再记录
    }
    this.points.push({x, y});
  }

  private redraw() {
    this.ctx.clearRect(0, 0, this.canvas.width, this.canvas.height);
    this.ctx.beginPath();
    this.points.forEach((p, i) => {
      if (i === 0) {
        this.ctx.moveTo(p.x, p.y);
      } else {
        this.ctx.lineTo(p.x, p.y);
      }
    });
    this.ctx.stroke();
  }

  // 绘画完成后，不需要保存Canvas内容
  // 只返回true表示"用户确认填满"
  confirm(): boolean {
    return this.points.length > 0;
  }

  clear() {
    this.points = [];
    this.ctx.clearRect(0, 0, this.canvas.width, this.canvas.height);
  }
}
```

**关键设计**：
- 限制笔触数量（200个点），防止内存溢出
- 绘画过程提供即时反馈
- 确认后只返回布尔值，不存储Canvas内容
- 缩略图用CSS统一填充效果

#### 2.7.6 移动端触摸交互优化（核心体验）

**背景**：用户反馈鼠标绘画体验不佳，手指触屏绘画更自然。本节重点解决移动端触摸交互问题。

**1. 触摸事件适配**

**问题**：Canvas 使用 `mousemove` 等鼠标事件，在手机上响应差或无法响应。

**解决方案**：同时监听鼠标和触摸事件，统一处理逻辑。

```typescript
class TouchCompatibleCanvas {
  private canvas: HTMLCanvasElement;
  private ctx: CanvasRenderingContext2D;
  private isDrawing = false;

  constructor(canvas: HTMLCanvasElement) {
    this.canvas = canvas;
    this.ctx = canvas.getContext('2d')!;
    this.setupEvents();
  }

  private setupEvents() {
    // 鼠标事件（桌面端）
    this.canvas.addEventListener('mousedown', (e) => this.startDrawing(e.offsetX, e.offsetY));
    this.canvas.addEventListener('mousemove', (e) => {
      if (this.isDrawing) this.draw(e.offsetX, e.offsetY);
    });
    this.canvas.addEventListener('mouseup', () => this.stopDrawing());
    this.canvas.addEventListener('mouseout', () => this.stopDrawing());

    // 触摸事件（移动端）- 关键：preventDefault 阻止滚动
    this.canvas.addEventListener('touchstart', (e) => {
      e.preventDefault(); // 阻止页面滚动
      const touch = e.touches[0];
      const rect = this.canvas.getBoundingClientRect();
      this.startDrawing(touch.clientX - rect.left, touch.clientY - rect.top);
    }, { passive: false }); // 必须设置 passive: false 才能调用 preventDefault

    this.canvas.addEventListener('touchmove', (e) => {
      e.preventDefault(); // 阻止页面滚动
      if (!this.isDrawing) return;
      const touch = e.touches[0];
      const rect = this.canvas.getBoundingClientRect();
      this.draw(touch.clientX - rect.left, touch.clientY - rect.top);
    }, { passive: false });

    this.canvas.addEventListener('touchend', () => this.stopDrawing());
    this.canvas.addEventListener('touchcancel', () => this.stopDrawing());
  }

  private startDrawing(x: number, y: number) {
    this.isDrawing = true;
    this.ctx.beginPath();
    this.ctx.moveTo(x, y);
  }

  private draw(x: number, y: number) {
    this.ctx.lineTo(x, y);
    this.ctx.stroke();
  }

  private stopDrawing() {
    this.isDrawing = false;
  }
}
```

**关键点**：
- `{ passive: false }` - 必须设置，才能调用 `preventDefault()`
- `e.preventDefault()` - **阻止页面滚动**，否则画画时页面会乱动
- `getBoundingClientRect()` - 正确计算触摸点在 Canvas 内的坐标
- 统一处理逻辑 - 鼠标和触摸事件共用同一套绘制逻辑

**2. 视口配置**

**问题**：手机浏览器默认会缩放页面，搞乱坐标计算和用户体验。

**解决方案**：在 HTML `<head>` 中添加视口配置。

```html
<meta name="viewport" content="width=device-width, initial-scale=1.0, maximum-scale=1.0, user-scalable=no">
```

**参数说明**：
- `width=device-width` - 宽度等于设备宽度，不缩放
- `initial-scale=1.0` - 初始缩放比例为 1.0
- `maximum-scale=1.0` - 禁止放大缩小
- `user-scalable=no` - 禁止用户手动缩放（双指捏合）

**为什么这样做**：
- 用户画画时不需要缩放
- 缩放会搞乱所有坐标计算
- 固定视口确保稳定的触摸体验

**7. 性能优化**

**问题**：手机性能有限，Canvas 绘画可能卡顿。

**解决方案**：优化绘画性能，减少不必要的重绘。

```typescript
class OptimizedCanvas {
  private rafId: number | null = null;
  private points: {x: number, y: number}[] = [];

  addPoint(x: number, y: number) {
    this.points.push({x, y});

    // 使用 requestAnimationFrame 批量绘制
    if (this.rafId === null) {
      this.rafId = requestAnimationFrame(() => {
        this.redraw();
        this.rafId = null;
      });
    }
  }

  private redraw() {
    // 批量绘制所有点
    this.ctx.beginPath();
    this.points.forEach((p, i) => {
      if (i === 0) {
        this.ctx.moveTo(p.x, p.y);
      } else {
        this.ctx.lineTo(p.x, p.y);
      }
    });
    this.ctx.stroke();
  }
}
```

| 用户主观判断 | 系统不判断任务是否完成，完全由用户决定什么时候画 |
| 及时反馈 | 弹出大格子（期待）→ 绘画过程（掌控）→ 填满确认（成就）→ 返回缩略图（满足） |
| 不可撤销 | 已填满的格子不可撤销，增强决策严肃性 |

> **注**：音效、震动反馈属于体验优化，留待后续迭代。MVP阶段聚焦核心交互体验。

### 4.3 技术评审总结（基于Linus Torvalds建议）

#### 4.3.1 做对的事

| ✅ 正确决策 | Linus评价 |
|-----------|----------|
| **本地存储优先** | "数据在用户设备上，不会因为服务器挂了丢数据。简单，可靠。" |
| **格子状态只存布尔值** | "布尔值就够了，别他妈存一堆无用数据。" |
| **时间记录与项目独立** | "简单的数据结构，十年后还能看懂。" |
| **MVP砍功能** | "先把能跑的东西做出来，再谈那些花里胡哨的。" |
| **已填满不可撤销** | "简单的状态机，不需要undo stack。" |

#### 4.3.2 关键技术决策

| 技术点 | 决策 | 理由 |
|-------|------|------|
| **时间精度** | 使用`performance.now()` | 不受系统时间调整影响，只向前走 |
| **Canvas实现** | 限制200个笔触点 | 防止内存溢出，不保存Canvas内容 |
| **数据结构** | 扁平JSON，无嵌套关联 | 十年后陌生人能维护 |
| **存储方式** | localStorage + JSON | 人类可读，文件锁防并发 |
| **平台选择** | 网页版（Web App + PWA） | 跨平台，快速迭代，无需审核，可安装到桌面 |
| **触摸交互** | 统一事件 + preventDefault | 一套代码跑遍所有设备，手指绘画体验佳 |
| **iOS适配** | 特殊处理（橡皮筋、安全区域） | 避免"能用但不爽"，确保完整体验 |

#### 4.3.3 工程原则

```
简单胜过聪明  →  数据结构扁平，避免过度抽象
显式胜过隐式  →  状态变化显式触发，不用魔法响应
无聊的代码是好代码 → 五分钟后还能看懂的代码
```

### 4.6 移动端开发注意事项（基于 Linus 建议）

**核心原则**：
> **"简单胜过聪明。一个代码库，跑遍所有设备。"**

**必做事项**：
1. ✅ 触摸事件必须用 `{ passive: false }` + `preventDefault()`
2. ✅ 视口必须禁止缩放（`user-scalable=no`）
3. ✅ iOS 安全区域必须适配（`env(safe-area-inset-*)`）
4. ✅ PWA manifest 必须配置完整（包括 iOS 专属 meta 标签）

**不要做的事**：
1. ❌ 不要为了移动端写两套代码
2. ❌ 不要用复杂的触摸库（如 Hammer.js），原生事件足够
3. ❌ 不要忽略 iOS 的特殊坑（橡皮筋、300ms 延迟）
4. ❌ 不要在移动端上臆测性能，必须真机测试

**测试方法**：
- Mac 用户：Safari → 开发菜单 → iPhone 模拟器
- 部署到 Vercel 后，用真机直接访问测试
- Chrome DevTools 设备模式仅供参考，不能替代真机
